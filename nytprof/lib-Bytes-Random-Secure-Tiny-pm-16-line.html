490=head1 NAME
491
492Bytes::Random::Secure::Tiny - A tiny Perl extension to generate
493cryptographically-secure random bytes.
494
495=head1 SYNOPSIS
496
497 use Bytes::Random::Secure::Tiny;
498
499 my $rng = Bytes::Random::Secure->new; # Seed with 256 bits.
500
501 my $bytes = $rng->bytes(32); # A string of 32 random bytes.
502 my $long = $rng->irand; # 32-bit random integer.
503 my $hex = $rng->hex_digits(10); # Ten hex digits.
504 my $string = $rng->string_from('abc', 10); Random string from a, b, & c.
505
506
507=head1 DESCRIPTION
508
509L<Bytes::Random::Secure> provides random bytes from a cryptographically
510secure random number generator (ISAAC), seeded from strong entropy sources
511on a wide variety of platforms. It is configurable, and has a flexible
512user interface.
513
514But it has a handful of dependencies. And its UI may be bigger than
515a typical user needs. L<Bytes::Random::Secure::Tiny> is designed to provide
516what 90% of Bytes::Random::Secure's users need, but with a simpler user
517interface, almost no configuration, and in a single module with no
518dependencies beyond core Perl.
519
520In many cases this module may be used as a drop-in replacement for
521L<Bytes::Random::Secure>. This module uses a cryptographic quality random
522number generator that uses the ISAAC algorithm, adapted from
523L<Math::Random::ISAAC>, and should be suitable for cryptographic purposes.
524The harder problem to solve is how to seed the generator. This module uses
525an approach adapted from L<Crypt::Random::Seed> to generate the initial
526seeds for the ISAAC CSPRNG.
527
528=head1 RATIONALE
529
530There are many uses for cryptographic quality randomness. This module aims to
531provide a generalized tool that can fit into many applications while providing
532a zero dependency chain, and a user interface that is both minimal and simple.
533You're free to come up with your own use-cases, but there are several
534obvious ones:
535
536=over 4
537
538=item * Creating temporary passphrases using the C<string_from> method.
539
540=item * Generating per-account random salt to be hashed along with passphrases
541to prevent rainbow table attacks.
542
543=item * Generating a secret that can be hashed along with a cookie's session
544content to prevent cookie forgeries.
545
546=item * Building raw cryptographic-quality pseudo-random data sets for testing
547or sampling.
548
549=item * Feeding secure key-gen utilities.
550
551=back
552
553Why use this module? This module employs several well-designed algorithms
554adapted from established CPAN tools to generate a strong random seed, and then
555to instantiate a high quality cryptographically secure pseudo-random number
556generator based on the seed. It has taken a good deal of research to come up
557with what I feel is a strong and sensible choice of established and published
558algorithms. The interface is designed with minimalism and simplicity in mind.
559
560Furthermore, this module runs its randomness through both statistical tests
561and NIST L<FIPS-140|https://en.wikipedia.org/wiki/FIPS_140> tests to verify
562integrity.
563
564As a C<::Tiny> module, the additional goals of low (or no) dependencies and a
565light-weight code base make this an ideal choice for environments where
566heavier dependency chains are problematic.

665=head2 FORK AND THREAD SAFETY
666
667When programming for parallel computation, create a unique
668C<Bytes::Random::Secure::Tiny> object within each process or thread.
669Bytes::Random::Secure::Tiny uses a CSPRNG, and sharing the same RNG between
670threads or processes will share the same seed and the same starting point. By
671instantiating the B::R::S::T object after forking or creating threads, a
672unique randomness stream will be created per thread or process.
673
674Always share the same RNG object between all non-concurrent consumers within
675a process, but never share the same RNG between threads or forked processes.

677=head2 STRONG RANDOMNESS
678
679It's easy to generate weak pseudo-random bytes. It's also easy to think you're
680generating strong pseudo-random bytes when really you're not. And it's hard to
681test for pseudo-random cryptographic acceptable quality. There are many high
682quality random number generators that are suitable for statistical purposes,
683but not necessarily up to the rigors of cryptographic use.
684
685Assuring strong (ie, secure) random bytes in a way that works across a wide
686variety of platforms is also challenging. A primary goal for this module is to
687provide cryptographically secure pseudo-random bytes while still meeting the
688secondary goals of simplicity, minimalism, and no dependencies. If more
689fine-grained control over seeding methods is needed, use
690L<Bytes::Random::Secure> instead.
691
692=head2 ISAAC
693
694The ISAAC algorithm is considered a cryptographically strong pseudo-random
695number generator. It has possible 1.0e2466 initial states. The best known
696attack for discovering initial state would theoretically take a complexity of
697approximately 4.67e1240, which has no practical impact on ISAAC's security.
698Cycles are guaranteed to have a minimum length of 2**40, with an average cycle
699of 2**8295. Because there is no practical attack capable of discovering
700initial state, and because the average cycle is so long, it's generally
701unnecessary to re-seed a running application. The results are uniformly
702distributed, unbiased, and unpredictable unless the seed is known.
703
704To confirm the quality of the CSPRNG, this module's test suite implements the
705L<FIPS-140-1|http://csrc.nist.gov/publications/fips/fips1401.htm> tests for
706strong random number generators. See the comments in C<t/27-fips140-1.t> for
707details.
708
709=head2 DEPENDENCIES
710
711In order to eliminate all non-core dependencies, this module inlines code
712adapted from L<Math::Random::ISAAC> and L<Crypt::Random::Seed>.
713
714The source of cryptographically secure pseudo-random data supplied by this
715module comes from the ISAAC algorithm. The ISAAC CSPRNG is seeded using
716algorithms adapted from C<Crypt::Random::Seed>. There are no known weaknesses
717in the ISAAC algorithm, and the algorithms adapted from Crypt::Random::Seed
718do a very good job of assuring the CSPRNG is well seeded.
719
720This module requires Perl 5.8 or newer. Unicode support in C<string_from> is
721best with Perl 5.8.9 or newer. See the INSTALLATION section in this document
722for details.
723
724=head2 BLOCKING ENTROPY SOURCE
725
726It is possible (and has been seen in testing) that the system's random
727entropy source might not have enough entropy in reserve to generate the seed
728requested by this module without blocking. In such cases, the blocking will
729time out after approximately two seconds, and seeding will fall back to
730a strong non-blocking source.
731

735string containing Unicode characters. For any modern Perl version, this
736will work just as you would hope. But some versions of Perl older than
7375.8.9 exhibited varying degrees of bugginess in their handling of Unicode.
738If you're depending on the Unicode features of this module while using Perl
739versions older than 5.8.9 be sure to test thoroughly, and don't be surprised
740when the outcome isn't as expected. ...this is to be expected. Upgrade.
741
742=head2 MODULO BIAS
743
744Care is taken so that there is no modulo bias in the randomness returned.
745As a matter of fact, this is exactly I<why> the C<string_from> method is
746preferable to a home-grown random string solution. However, the algorithm to
747eliminate modulo bias can impact the performance of the C<string_from>
748method. Any time the length of the bag string is significantly less than the
749nearest greater or equal factor of 2**32, performance will degrade.
750Unfortunately there is no known algorithm that improves upon this situation.
751Fortunately, for sanely sized strings, it's a minor issue. To put it in
752perspective, even in the case of passing a "bag" string of length 2**31
753(which is huge), the expected time to return random bytes will only double.
